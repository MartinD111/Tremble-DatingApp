# Tremble - Google Play Console Data Safety Questionnaire

Use the answers below when filling out the **Data Safety form** in the Google Play Console for the Tremble mobile application.

---

## 🔒 Section 1: Data Collection & Security Practices

### 1. Does your app collect or share any of the required user data types?
* **Answer:** Yes

### 2. Is all of the user data collected by your app encrypted in transit?
* **Answer:** Yes (All data is transmitted securely over HTTPS/TLS).

### 3. Do you provide a way for users to request that their data be deleted?
* **Answer:** Yes (Users can request account deletion in-app or via the web form at [trembledating.com/erasure](https://trembledating.com/erasure)).

---

## 📂 Section 2: Data Types Collection & Usage

For each data type declared below, answer the follow-up questions in the console.

### 📍 Location

#### 1. Approximate Location (Geohash)
* **Collected:** YES
* **Shared:** NO (Processed privately on Firebase/Upstash servers)
* **Ephemeral (Processed in memory/RAM only):** NO (Geohash precision 7 cell is temporarily stored in Firestore with a strict 2-hour TTL expiration policy)
* **Linked to user:** YES (Associated with the user's profile for proximity detection)
* **Used for tracking purposes:** NO
* **Required / Optional:** Required (Core functionality of the proximity Radar)
* **Purpose:** App functionality (proximity detection & matching)
* **Declaration Detail:** Tremble collects a coarse geohash block (precision 7, representing ~75m × 75m area) to determine matches.

#### 2. Precise Location (GPS Coordinates)
* **Collected:** NO
* **Declaration Detail:** Exact GPS coordinates are processed entirely in Cloud Function RAM (memory) to evaluate Safe Zones and proximity. They are never saved, persisted, or stored on our servers.

## 🤝 Section 3: Data Sharing & Third-Party Processors

Select **NO** to general third-party sharing in the Play Console since user data is never sold or shared for external advertising/telemetry. However, document the following operational sub-processors used strictly to deliver the service:

| Provider | Purpose | Data Transmitted |
| --- | --- | --- |
| **Firebase / Google Cloud** | Infrastructure, Database, Auth | User Profile, approximate geohash, authentication details. Encrypted at rest & in transit. |
| **Cloudflare R2** | Media Storage | User profile photos only. Host location: media.trembledating.com (public GET only, list disabled). |
| **RevenueCat** | Subscription & Billing Engine | Purchase tokens, subscription lifecycle states (no payment/card info). |
| **Resend** | Transactional Emails | Emails sent from info@trembledating.com. |
| **Upstash Redis** | High-performance cache | Transient proximity signals during active scan (TTL-gated). |

---

## 🛡️ Section 4: Security & Deletion Practices

* **Encryption in Transit:** All network traffic is encrypted using HTTPS/TLS protocols.
* **Encryption at Rest:** Firestore, Authentication, Cloudflare R2, and Upstash data are fully encrypted at rest using infrastructure-level advanced encryption standards (AES-256).
* **Data Deletion:** Users can instantly trigger account deletion under GDPR Article 17 (Right to Erasure) in-app or via [trembledating.com/erasure](https://trembledating.com/erasure). This deletes their auth record, Firestore profile, and photos from Cloudflare R2 immediately.
