**File:** `functions/src/modules/gdpr/gdpr.functions.ts`

- Slice `refs` into chunks of 500

### Step 2b — Fix `deleteUserAccount` collection queries

**File:** `functions/src/modules/gdpr/gdpr.functions.ts`

⚠️ **FOUNDER DECISION POINT — `reports` collection:**

Reports documents have `reporterId` (who filed the report) and `reportedId` (who was reported).
GDPR Art. 17 Right to Erasure vs. Art. 17(3)(e) exemption for legal defence.

**Two options:**
- **Option A (Full erase):** Delete reports where `reporterId == uid` AND `reportedId == uid`.
  Fully complies with Art. 17. Destroys moderation evidence.
- **Option B (Anonymise):** Delete reports where `reporterId == uid` (user's own reports).
  For reports where `reportedId == uid`: anonymise by replacing `reportedId` with `"[deleted]"`
  rather than deleting the document. Preserves moderation evidence under Art. 17(3)(e).

**Recommended: Option B.** This is the standard approach for dating/social apps.
Founder must confirm before this step is implemented.

Replace the body of `deleteUserAccount` batch block with sequential `deleteBatch` calls:

| # | Collection | Query | Notes |
|---|---|---|---|
| 1 | `users` | doc `uid` | direct delete |
| 2 | `proximity` | doc `uid` | direct delete |
| 3 | `waves` | `fromUid == uid` | paginated via deleteBatch |
| 4 | `waves` | `toUid == uid` | paginated via deleteBatch |
| 5 | `proximity_events` | `uid == uid` | paginated via deleteBatch (verify field name in proximity.functions.ts before writing) |
| 6 | `proximity_notifications` | `.where("users", "array-contains", uid)` | array-contains query — NOT a uid field! paginated via deleteBatch |
| 7 | `idempotencyKeys` | `__name__ >= {uid}:` range | paginated via deleteBatch |
| 8 | `reports` | `reporterId == uid` → delete; `reportedId == uid` → **Option A: delete / Option B: anonymise** | await founder decision |
| 9 | `matches` | `userA == uid` | paginated via deleteBatch |
| 10 | `matches` | `userB == uid` | paginated via deleteBatch |
| 11 | `rateLimits` | `__name__ >= {uid}:` range | paginated via deleteBatch |

Then R2 deletion and Firebase Auth deletion remain unchanged.

Update the JSDoc comment to list all collections correctly.

**Verification:** `cd functions && npm test -- --testPathPattern=gdpr`
→ Cases 1, 2, 3 should now pass. Case 4 still fails.

## 4. RISKS & TRADEOFFS

**Risk 1 — `reports` deletion strategy (founder decision required before Step 2b)**
GDPR Art. 17 vs. Art. 17(3)(e) exemption. See Step 2b for options. Implementing before
a decision is made means either violating GDPR or destroying safety evidence.

**Risk 2 — `proximity_events` uid field name**
Must verify exact field name in `proximity.functions.ts` before writing the query.
The rule `allow write: if request.auth != null` provides no hint. A wrong field name
silently returns 0 results with no error.

**Risk 3 — `proximity_notifications` uses array-contains, not uid field**
Confirmed: documents have `users: [fromUid, toUid]` array (proximity.functions.ts line ~344).
Query must be `.where("users", "array-contains", uid)`. A uid-field query would silently
return 0 results.

**Risk 4 — Batch pagination latency**
Sequential `deleteBatch` calls add latency. For a typical user: imperceptible. For
a power user with thousands of proximity_events: 10–30s. Acceptable for GDPR deletion
(no SLA on this endpoint).

**Risk 5 — `idempotencyKeys` key format**
Keys are document IDs with format `{uid}:{action}`. Confirm this by checking how
`idempotencyKeys` are written in the functions before implementing the range query.

**Debt introduced:** None. Pure correction of existing functionality.
